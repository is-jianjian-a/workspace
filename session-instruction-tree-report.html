✅ 成功模式（要保留）

1. 示例驱动约束（Example-Driven Constraint）

关键帧： Prompt v2 提供了完整的TOP1/TOP2/TOP3示例格式
效果： 模型从"抽象理解"转为"模式匹配"，输出精确符合期望
本质： 用少样本提示(Few-shot)替代自然语言描述，消除歧义

2. 否定+重构的修正模式

关键帧： "把P0、P1那些去掉"（否定旧格式）+ "按照UGC条数来排序"（给出新规则）+ "类似下面："（给出示例）
效果： 一次性完成"去旧+立新+示范"三重约束
本质： 否定必须伴随替代方案，否则模型会自由发挥

3. 用户原声视角的语言锚定

关键帧： "尽可能使用用户原声的视角来描述"
效果： 需求描述从"功能视角"（提供情绪识别）转为"用户视角"（备孕反复失败时有人能懂）
本质： 语言风格约束比内容约束更根本，决定了输出的"气味"

❌ 失败模式（要剔除）

1. 抽象描述陷阱

错误： "把产品机会点、具体需求、总互动量、相关原声使用等合成一个HTML报告展示"
问题： 只描述了"要什么"，没描述"不要什么"和"像什么样子"
后果： 模型沿用历史session的v3报告模板（带P0/P1标签、功能视角、详细分析），与用户期望的紧凑格式偏离
教训： 抽象描述 = 模型自由发挥 = 路径依赖错误

2. 隐式上下文依赖

错误： 假设模型会"理解"当前session的上下文状态
问题： 当前session实际继承的是历史session（ai-demand-research-v3）的方法论和模板
后果： 模型输出的是历史项目的"完整分析报告"格式，而非用户当前想要的"紧凑机会点列表"
教训： 跨session上下文是双刃剑，必须显式覆盖或重置

💡 关键认知

成功不是"写对一句话"，而是"控制了模型的搜索空间"

本session的v1→v2转变验证了这一点：
• v1 给了目标（合成报告）→ 模型搜索空间大 → 输出偏离
• v2 给了目标+否定+示例 → 搜索空间被三重约束压缩 → 输出精确匹配

最有效的prompt结构：
1. 否定旧模式（"去掉P0/P1"）
2. 建立新规则（"按UGC条数排序"）
3. 提供具体示例（"类似下面：TOP1..."）
4. 语言风格锚定（"用户原声的视角"）
