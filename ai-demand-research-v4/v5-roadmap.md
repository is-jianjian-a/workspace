## 一、规划背景

v4.0 是最小可行改进集，聚焦三大根本问题（JTBD框架、Kano+RICE优先级、假设验证闭环）。v5.0 将在 v4.0 验证有效的基础上，引入更深层次的改进，将工作流从"优秀的数据处理引擎"升级为"完整的需求发现与验证系统"。

**v4 → v5 的跃迁逻辑**：
- v4 解决"洞察是否真实"和"优先级是否可靠"
- v5 解决"洞察是否全面"和"体系是否可持续"

---

## 二、v5 改进总览

v5 包含 v3 优化建议报告中剩余的 **4 个改进维度**（原P1+P2）：

| 改进维度 | 原编号 | 优先级 | 核心内容 | 预期收益 |
|---------|--------|--------|---------|---------|
| **多源三角验证** | 改进点2 | P1 | UGC+行为数据+深度访谈三层验证 | 洞察置信度从"可能"提升至"极可能" |
| **用户旅程地图** | 改进点6 | P1 | 从静态标签升级为动态旅程分析 | 识别断点与情绪低谷，指导产品优化 |
| **持续发现模式** | 改进点1 | P2 | 批处理→持续运行的数据管道 | 需求发现从"周级"缩短至"天级" |
| **North Star Metric** | 改进点7 | P2 | 结果导向的衡量体系 | 从"输出报告数"转向"改变决策数" |

此外，v5 还将引入 **竞品评论采集** 作为 UGC 数据源的扩展。

## 三、改进详解

### 3.1 多源三角验证体系（P1）

#### 问题本质
v4 仅依赖 UGC 文本数据，存在**幸存者偏差**——只能捕捉到愿意且能够公开表达的用户声音。不同平台的算法推荐机制会系统性扭曲内容分布，高互动内容不代表高真实需求。

#### 改进方案
构建三层数据源体系：

```
┌─────────────────────────────────────────────────────────────┐
│                    三层数据源体系                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  L1: 公开声音 (UGC)          ← v4已有，v5扩展              │
│     ├── 小红书、B站、知乎、酷安                              │
│     ├── App Store / Google Play 评论  ← v5新增             │
│     └── G2 / Capterra (B端产品)  ← v5新增                  │
│                                                             │
│  L2: 行为数据  ← v5全新引入                                 │
│     ├── 产品埋点（页面浏览、点击、停留）                      │
│     ├── 用户行为日志（功能使用频率、路径）                    │
│     └── 转化漏斗（注册→激活→留存→付费）                     │
│                                                             │
│  L3: 深度对话  ← v5全新引入                                 │
│     ├── 用户访谈（5-8次半结构化访谈）                        │
│     ├── 可用性测试（原型测试）                               │
│     └── Switch Interview（JTBD深度访谈）                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 三角验证实施逻辑
1. **L1 发现信号**：从 UGC 数据中发现高互动痛点/期望信号
2. **L2 验证行为**：检查产品埋点数据，确认用户是否在实际行为中表现出相同模式
3. **L3 探索动机**：对关键信号进行 5-8 次深度访谈，探索行为背后的 JTBD

> **原则**：当三个层级的数据指向同一结论时，洞察的置信度从"可能"提升至"极可能"。

#### 实施要点
- 在产品中部署事件追踪系统（如 Amplitude、Mixpanel、神策数据）
- 建立用户访谈招募池，保持 20-30 名可用受访者的储备
- 为每个 P0 机会建立"三角验证清单"，要求至少两个层级的数据支撑

#### 对工作流的影响
新增步骤（插入到 v4 的适当位置）：

| 新步骤 | 位置 | 说明 |
|--------|------|------|
| 3.5 行为数据采集 | 步骤3之后 | 从埋点系统导出行为数据 |
| 3.6 竞品评论采集 | 步骤3之后 | App Store/Google Play/G2/Capterra |
| 9.5 深度访谈 | 步骤9之后 | 对 Top 机会进行访谈验证 |
| 13.5 三角验证报告 | 步骤13之后 | 汇总三层数据的验证结论 |

### 3.2 用户旅程地图（Customer Journey Map, CJM）（P1）

#### 问题本质
v4 的用户画像基于静态标签（身份、设备、使用深度），无法反映用户的动态变化和在旅程中的痛点分布。

#### 改进方案
引入用户旅程地图，描绘用户从"初次认知"到"忠诚推荐"的完整体验路径：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        用户旅程地图（CJM）                                    │
├─────────────┬─────────────┬─────────────┬─────────────┬─────────────────────┤
│   认知       │   考虑       │   首次使用   │   日常使用   │   忠诚/流失         │
│  Awareness  │Consideration│  Onboarding │ Engagement  │ Loyalty/Churn       │
├─────────────┼─────────────┼─────────────┼─────────────┼─────────────────────┤
│ 用户如何第一 │ 用户如何评估 │ 用户如何完成 │ 用户的使用   │ 哪些信号预示        │
│ 次听说产品？ │ 产品？       │ 第一次成功   │ 习惯是什么？ │ 用户可能流失？      │
│             │             │ 体验？       │             │                     │
├─────────────┼─────────────┼─────────────┼─────────────┼─────────────────────┤
│ 数据需求：   │ 数据需求：   │ 数据需求：   │ 数据需求：   │ 数据需求：          │
│ 搜索词分析   │ UGC评论      │ 注册漏斗     │ DAU/MAU     │ 使用频率下降        │
│ 流量来源     │ 竞品对比行为 │ 激活事件     │ 功能使用分布 │ 负面反馈            │
└─────────────┴─────────────┴─────────────┴─────────────┴─────────────────────┘
         ↑              ↑              ↑              ↑              ↑
       触点          触点           触点           触点           触点
      （广告/      （UGC/        （引导页/       （推送/        （客服/           
       口碑）       评测）         教程）         社区）         调研）
```

在每个阶段标注：
- **用户行动**：用户在此阶段做什么
- **触点**：用户与产品的接触点
- **情绪曲线**：用户的情感变化（高/中/低）
- **痛点**：此阶段的主要痛点（关联 Job Stories）
- **机会**：此阶段的改进机会

#### 实施要点
- 为产品核心用户旅程绘制 CJM，标注当前已知痛点
- 建立用户分层模型（如 RFM：Recency-Frequency-Monetary）
- 在需求分析时，不仅分析"哪些需求高频"，还分析"哪些用户群体的哪些需求未被满足"
- 将用户分层数据与 UGC 分析结果交叉验证

#### 对工作流的影响
- **步骤7升级**：用户画像 → 用户旅程地图
- **新增输出**：`07-journey/customer_journey_map.html`（可视化旅程地图）
- **报告增强**：最终分析报告中增加"旅程阶段筛选"功能

### 3.3 持续发现模式（P2）

#### 问题本质
v4 是"批处理式项目运行"——每轮调研从零开始，采集数据、分析、输出报告后结束。这种批处理模式存在三个根本性缺陷：
1. 无法捕捉需求的动态演变
2. 每轮调研的沉没成本高
3. 团队与用户的连接在调研结束后断裂

#### 改进方案
引入持续发现框架，将需求调研从"项目"转变为"产品运营习惯"：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        持续发现模式架构                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                     持续运行的数据管道                                │   │
│  │  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐     │   │
│  │  │ UGC爬虫   │ → │ 数据清洗  │ → │ Job提取   │ → │ 信号监控  │     │   │
│  │  │ (定时)    │    │ (自动)    │    │ (自动)    │    │ (看板)    │     │   │
│  │  └──────────┘    └──────────┘    └──────────┘    └──────────┘     │   │
│  │         ↑                                              ↓           │   │
│  │         └────────────── 每周回顾 ───────────────────────┘           │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                              ↓                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                     季度深度调研（保留v4模式）                        │   │
│  │  探索"未知领域"的新主题，进行完整的16步分析                          │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 模式对比

| 维度 | v4 批处理模式 | v5 持续发现模式 |
|------|-------------|---------------|
| 调研频率 | 项目触发，间隔数周/数月 | 每周固定运行，自动化监控 |
| 数据更新 | 一次性采集 | 增量采集+实时清洗 |
| 机会追踪 | 静态报告 | 动态机会看板 |
| 团队参与 | 研究员主导 | 产品三人组每周参与 |
| 适用场景 | 探索新主题 | 监控已知主题的演变 |

#### 实施要点
1. 建立 **UGC 数据管道**：将数据采集从手动触发改为定时自动运行，支持增量更新
2. 搭建 **机会监控看板**：使用 Notion/Figma 等工具创建动态的机会追踪面板
3. 设定 **每周发现节奏**：每周五下午固定 30 分钟，产品三人组回顾本周新发现的需求信号
4. 保留 **季度深度调研**：持续发现处理"已知领域"的增量信号，季度深度调研探索"未知领域"

#### 对工作流的影响
- **新增模式**："快速监控模式"（每周运行，只执行步骤3-5-9）
- **新增工具**：`scripts/monitor.py`（定时监控脚本）
- **新增输出**：`00-meta/opportunity_dashboard.html`（动态机会看板）

### 3.4 North Star Metric（P2）

#### 问题本质
v4 的成果衡量是**输出导向的**（采集了多少数据、提取了多少信号、交付了多少报告），而非**结果导向的**（调研结果是否改变了产品决策、上线功能是否提升了用户价值）。

#### 改进方案
引入 North Star Metric（北极星指标）作为需求调研工作的核心衡量标准：

> **"经调研验证的需求，其上线功能的 30 日用户采纳率"**

#### 指标分解树

```
                    ┌─────────────────────────────┐
                    │   North Star Metric         │
                    │  验证需求采纳率              │
                    │   (目标: ≥60%)              │
                    └──────────────┬──────────────┘
                                   │
           ┌───────────────────────┼───────────────────────┐
           │                       │                       │
    ┌──────▼──────┐        ┌──────▼──────┐        ┌──────▼──────┐
    │ 假设验证     │        │ 洞察到决策   │        │ 上线功能     │
    │ 通过率       │        │ 转化率       │        │ 满意度       │
    │ (目标: ≥70%)│        │ (目标: ≥50%)│        │ (目标: ≥4.0) │
    └─────────────┘        └─────────────┘        └─────────────┘
```

#### 实施要点
1. 为需求调研团队设定明确的 North Star Metric 目标
2. 建立"需求全生命周期追踪"机制，从需求发现到功能上线再到效果评估全程可追踪
3. 每季度回顾 North Star Metric 的达成情况，识别瓶颈环节
4. 将 North Star Metric 与 OKR 体系对接，确保调研目标与业务目标对齐

#### 对工作流的影响
- **新增步骤**：步骤 16.5 "效果追踪"（功能上线后 30 天评估）
- **新增输出**：`16-reports/north_star_dashboard.html`
- **报告增强**：改进建议报告中增加 North Star Metric 达成分析

### 3.5 竞品评论采集（v5 新增）

#### 背景
App Store / Google Play 评论是 VOC（Voice of Customer）富矿，天然包含：
- 功能需求（"希望能增加XX功能"）
- 情感倾向（评分 + 评论内容）
- 版本关联（"更新后出现了XX问题"）

#### 数据源扩展

| 平台 | 类型 | 价值 | 采集方式 |
|------|------|------|---------|
| App Store | 应用商店 | iOS用户反馈 | Playwright + 指纹伪装 |
| Google Play | 应用商店 | Android用户反馈 | Playwright + 指纹伪装 |
| G2 | B端评测 | 企业用户反馈 | API / 爬虫 |
| Capterra | B端评测 | 软件对比评价 | API / 爬虫 |

#### 对工作流的影响
- **步骤3扩展**：数据源增加竞品评论平台
- **新增脚本**：`step03_crawl_competitor_reviews.py`
- **分析增强**：竞品评论纳入 Job Story 提取和 Kano 分类

## 四、v5 工作流架构（预测）

基于以上改进，v5 的工作流可能从 v4 的 **16 步扩展至 22 步**：

```
阶段一：感知 (SENSING) - 6步
  1. 项目初始化
  2. 搜索词设计
  3. UGC数据采集
  3.5 竞品评论采集 ← 新增
  3.6 行为数据采集 ← 新增
  4. 数据清洗与融合 ← 增强（多源融合）

阶段二：理解 (UNDERSTANDING) - 7步
  5. Job Story提取
  6. Job地图绘制
  7. 用户旅程地图 ← 升级（原用户画像）
  8. Kano分类
  9. RICE评分
  9.5 深度访谈 ← 新增
  9.6 三角验证报告 ← 新增

阶段三：验证 (VALIDATION) - 4步
  10. 假设形成
  11. 实验设计
  12. 快速验证
  13. 学习闭环

阶段四：交付 (DELIVERY) - 5步
  14. 洞察生成
  15. 搜索词优化
  16. 三件套交付
  16.5 效果追踪 ← 新增
  17. 持续发现更新 ← 新增（仅持续发现模式）
```

## 五、实施路线图

建议分三个阶段实施 v5：

### Phase 1：数据源扩展（第1-2个月）
- 部署竞品评论采集（App Store / Google Play）
- 建立用户访谈招募池
- 绘制用户旅程地图 v1.0

**预期产出**：
- 竞品评论数据库
- 用户访谈 SOP
- 用户旅程地图 HTML

### Phase 2：三角验证闭环（第3-4个月）
- 部署产品埋点，启动行为数据追踪
- 对 v4 的 Top 机会进行深度访谈验证
- 建立三角验证报告模板

**预期产出**：
- 行为数据看板
- 三角验证报告模板
- 验证后的需求清单（Confidence ≥ 80%）

### Phase 3：持续发现体系（第5-6个月）
- 将 UGC 采集改造为持续运行的数据管道
- 建立 North Star Metric 追踪体系
- 建立动态机会监控看板

**预期产出**：
- 持续发现仪表板
- North Star Metric 基线
- 完整的 v5 工作流文档

## 六、v5 边界说明

### v5 包含
- ✅ 多源三角验证（UGC + 行为数据 + 深度访谈）
- ✅ 用户旅程地图（CJM）
- ✅ 竞品评论采集（App Store / Google Play / G2 / Capterra）
- ✅ 持续发现模式（数据管道 + 动态看板）
- ✅ North Star Metric（结果导向衡量）
- ✅ 效果追踪（功能上线后评估）

### v5 不包含（留到v6+）
- ❌ 机会解决方案树（Opportunity Solution Tree）
- ❌ 设计思维后三阶段（构思/原型/测试的完整集成）
- ❌ AI 辅助访谈（自动进行用户访谈）
- ❌ 预测性需求发现（基于趋势预测未来需求）

## 七、决策检查点

在实施 v5 之前，需要确认以下问题：

| 检查项 | 问题 | 决策标准 |
|--------|------|---------|
| v4 验证 | v4 的 JTBD + Kano/RICE + 假设闭环是否有效？ | 运行至少 2 轮完整调研 |
| 资源投入 | 是否有预算部署埋点系统和访谈资源？ | 确认技术+人力预算 |
| 组织支持 | 产品团队是否愿意参与假设验证？ | 产品团队承诺参与 |
| 数据合规 | 行为数据采集是否符合隐私政策？ | 法务确认合规 |
